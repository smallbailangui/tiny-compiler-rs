## 二、数据结构设计（约 1 分 30 秒）

> *展示：打开 `src/lab2/types.rs`，滚动浏览第 1-113 行*

### 2.1 文法符号类型

首先看文法符号的枚举定义（第 7-12 行）：

```rust
pub enum SymbolType {
    Terminal,    // 终结符
    NonTerminal, // 非终结符

```

与指导书中的 `SymbolType` 枚举完全对应。

### 2.2 GrammarSymbol 与 Production

`GrammarSymbol` 结构体（第 14-22 行）包含了指导书要求的全部字段：
- `name`：符号名称
- `sym_type`：符号类别（对应指导书的 `SymbolType`）
- `productions`：该非终结符的所有产生式编号列表（对应 `pProductionTable`）
- `first_set`：FIRST 集（对应 `pFirstSet`）
- `follow_set`：FOLLOW 集（对应 `pFollowSet`）
- `dependent_in_follow`：FOLLOW 依赖集（对应 `pDependentSetInFollow`）

这里我将终结符和非终结符合并为一个结构体，通过 `sym_type` 字段区分，减少了代码冗余。

`Production` 结构体（第 24-36 行）包含：
- `production_id`：产生式序号（对应 `productionId`）
- `head`：产生式头
- `body`：产生式体中的文法符序列（对应 `pBodySymbolTable`）

### 2.3 LR(0) 相关数据结构

`LR0Item`（第 48-52 行）由 `production_id` 和 `dot_position` 两个字段唯一确定一个 LR(0) 项目，实现了 `Ord` 和 `Hash` trait，可以直接放入 `BTreeSet` 中去重排序。

`ItemSet`（第 54-69 行）包含：
- `state_id`：状态序号
- `core_items`：核心项集合（对应 CORE 类别）
- `items`：完整项集合（核心项 + 非核心项），对应指导书的 `pItemTable`

这里我用 `BTreeSet` 分别存储核心项和完整项，核心项用于判断项集是否相同——这是子集构造法中新旧项集判等的关键。

`TransitionEdge`（第 71-76 行）和 `LR0DFA`（第 78-83 行）对应指导书中的变迁边和 DFA 定义。

### 2.4 语法分析表数据结构

`ActionCategory` 枚举有三种取值：`Shift`（对应 's' 移入）、`Reduce`（对应 'r' 规约）、`Accept`（对应 'a' 接受）。

`ActionCell` 包含 `action_type` 和 `id`，`GotoCell` 包含 `next_state`。

`LRParseTable` 使用 `BTreeMap<(usize, String), ActionCell>` 存储 ACTION 表。正如指导书所建议的，使用字典类型有利于提升查找性能。Rust 的 `BTreeMap` 相比 `HashMap` 的优势在于键有序，方便按状态编号排列输出。

## 三、FIRST 函数的实现（约 1 分钟）

> *展示：打开 `src/lab2/first.rs`，滚动浏览第 5-70 行*

### 3.1 `first_of_sequence` —— 序列的 FIRST 函数

这个函数（第 5-39 行）对应指导书任务 1.1"产生式的 FIRST 函数求解"。

核心逻辑分三步：
1. 空序列直接返回 `{ε}`。
2. 遍历文法符号序列，将每个符号 FIRST 集中**非 ε** 的符号加入结果。
3. 如果当前符号的 FIRST 集包含 ε，继续看下一个符号；否则停止。
4. 若所有符号都能推导出 ε，最终结果也加入 ε。

举个例子：对于产生式体 `[T, S]`，假设 `FIRST(T) = {(, ε}`，`FIRST(S) = {(, ), ε}`，那么 `first_of_sequence([T, S]) = {(, ), ε}`——因为 T 可以推导 ε，所以继续看 S，S 的 `(` 和 `)` 都要加进来，两者都能推导 ε，所以 ε 也加进来。

### 3.2 `compute_first_sets` —— 不动点迭代

这个函数（第 41-70 行）采用**不动点迭代算法**：在 `loop` 循环中，每轮对每条产生式调用 `first_of_sequence` 计算新的 FIRST 符号，同时插入到产生式的 `first_set` 和非终结符的 `first_set` 中。当某轮没有新符号加入时（`changed == false`），迭代收敛，算法终止。

这种同时更新产生式和非终结符 FIRST 集的设计，一次性完成了指导书任务 1.1 和 1.2。

## 四、FOLLOW 函数的实现（约 50 秒）

> *展示：打开 `src/lab2/follow.rs`，滚动浏览第 4-74 行*

FOLLOW 集同样采用**不动点迭代**，严格遵循三条经典规则：

**规则 1（初始化，第 7-18 行）：** 对增广开始符号 `S'` 和原始开始符号 `S`，将结束标记 `$` 插入 FOLLOW 集。

**规则 2（FIRST 传递，第 47-54 行）：** 对产生式 `A → αBβ`，将 `FIRST(β) \ {ε}` 加入 `FOLLOW(B)`。这里是取了 `β` 子序列（第 42 行 `let beta = &body[(i + 1)..]`），调用 `first_of_sequence` 获取其 FIRST 集。

**规则 3（FOLLOW 传递，第 57-66 行）：** 若 `β` 为空或 `FIRST(β)` 包含 ε，将 `FOLLOW(A)` 加入 `FOLLOW(B)`。同时通过 `dependent_in_follow` 字段记录 B 依赖 A，这个字段直接对应指导书中的 `pDependentSetInFollow`。

这里有一个巧妙的设计：我不需要在规则 3 中再次迭代已加入的 FOLLOW 符号，因为在最外层 `loop` 已经保证了多次迭代直到收敛。每一轮都会基于当前所有产生式重新计算，自然就能把依赖传递到位。

## 五、LR(0) 项集闭包与 DFA 构造（约 1 分 30 秒）

> *展示：打开 `src/lab2/lr0.rs`，逐函数讲解*

### 5.1 `get_closure` —— 闭包求解（第 5-27 行）

这个函数对应指导书任务 2.1 `void getClosure(ItemSet* itemSet)`。

核心算法：
1. 用核心项初始化一个 `VecDeque` 工作队列。
2. 循环出队，检查圆点后下一个符号是否为非终结符。
3. 如果是非终结符 `B`，将 `B` 的所有产生式（圆点在开头，即 `dot_position = 0`）作为新项插入。
4. `BTreeSet::insert` 返回 `true` 表示是新项，将其入队继续扩展；返回 `false` 表示已存在，跳过。

举例说明：对于算术表达式文法，初始核心项是 `[E' → ·E]`。闭包求解过程：
- 圆点后是 `E`，E 有产生式 `E → E+T` 和 `E → T`，加入 `[E → ·E+T]` 和 `[E → ·T]`。
- 看 `[E → ·T]`，圆点后是 `T`，加入 `[T → ·T*F]` 和 `[T → ·F]`。
- 继续扩展，最终 I₀ 包含 6 个项目。

### 5.2 `exhaust_transitions` —— 变迁穷举（第 29-49 行）

对应指导书任务 2.2 `void exhaustTransition(ItemSet* itemSet)`。

遍历项集的所有项目，检查圆点后是否有符号。按驱动符分组，生成"圆点前进一位"的新核心项。例如在 I₀ 中，`[E' → ·E]` 驱动符是 `E`，生成 `[E' → E·]`；`[E → ·E+T]` 驱动符也是 `E`，生成 `[E → E·+T]`。这两个新核心项都归入驱动符 `E` 的分组。

返回 `BTreeMap<String, BTreeSet<LR0Item>>`——键是驱动符，值是目标核心项集合。

### 5.3 `build_lr0_dfa` —— DFA 构造（第 52-101 行）

对应指导书任务 2.3"文法的 LR(0) 型 DFA 求解"。

完整实现了**子集构造法**：
1. 从增广产生式 `S' → S`（id=0, dot=0）构造初始核心项集。
2. 调用 `get_closure` 求 I₀ 的闭包。
3. 维护一个索引 `idx` 遍历已生成的项集。对每个项集调用 `exhaust_transitions` 穷举变迁。
4. 对每个驱动符的目标核心项集合，判断是否与已有项集的核心项相同——通过 `BTreeSet` 的判等实现（第 76-78 行）。
5. 若是新项集，创建并调用 `get_closure`；若已存在则复用状态编号。
6. 创建 `TransitionEdge` 记录变迁边。

这个算法保证 DFA 状态数最小——因为通过核心项判等可以合并等价状态。

## 六、SLR(1) 判定与 LR 语法分析表（约 1 分 10 秒）

> *展示：打开 `src/lab2/slr.rs`*

### 6.1 `check_slr1` —— SLR(1) 判定（第 6-77 行）

对应指导书任务 2.4。

对每个项集分别检测两类冲突：
- **移入-规约冲突**：规约项的 FOLLOW 集与移入终结符有交集。
- **规约-规约冲突**：两条不同规约产生式的 FOLLOW 集相交。

以算术表达式文法为例，程序输出 `✔ 该文法是 SLR(1) 文法`，说明无冲突。这是因为运算符 `+` 和 `*` 的优先级使得 FOLLOW 集天然分离——例如在包含 `E → E·+T` 和 `T → T*F·` 的项集中，规约项 `T*F` 的 FOLLOW 集是 `{+, $, )}`，而移入终结符也是 `+`，这里就需要看 `+` 的优先级——SLR(1) 通过 FOLLOW 集可以正确处理。

### 6.2 `build_lr_parse_table` —— 填写语法分析表（第 79-169 行）

对应指导书任务 2.5。

**ACTION 表填写**遵循 SLR(1) 规则：
- 终结符驱动的变迁边 → `Shift(to_state)`，记为 `s{状态号}`
- 规约项 `A → α·` → 对 FOLLOW(A) 中每个终结符填入 `Reduce(production_id)`，记为 `r{产生式号}`
- 增广产生式规约项 `S' → S·` → `ACTION[state, $] = Accept`，记为 `acc`

**GOTO 表填写**：非终结符驱动的变迁边，填入 `next_state`，直接写状态号。

表格使用 `BTreeMap` 存储，输出时按状态编号和符号名自动排序，方便查阅。

## 七、LL(1) 语法分析表（约 40 秒）

> *展示：打开 `src/lab2/ll1.rs`*

### 7.1 `build_ll1_parse_table`（第 6-42 行）

对每条产生式 `A → α`：
1. 计算 `FIRST(α)`。
2. 对每个终结符 `t ∈ FIRST(α) \ {ε}`，在 `M[A, t]` 中加入该产生式。
3. 若 `ε ∈ FIRST(α)`，对每个终结符 `f ∈ FOLLOW(A)`，也在 `M[A, f]` 中加入该产生式。

### 7.2 `check_ll1`（第 45-55 行）

遍历分析表，若某表项包含多于一条产生式，则报告冲突——该文法不是 LL(1) 文法。

两个 LL(1) 测试用例都无冲突，验证了实现的正确性。

## 附：常见提问与回答（Q&A 准备）

**Q1：为什么核心项判等用 `BTreeSet` 而不是逐项比较？**

A：`BTreeSet` 要求元素实现 `Ord + Eq`，`LR0Item` 同时实现了它们。直接用 `==` 比较两个 `BTreeSet<LR0Item>`，Rust 会自动逐元素比较，既简洁又不会出错。相比手动写循环比较，代码量少且更可读。

**Q2：FOLLOW 计算中 `dependent_in_follow` 字段的作用？**

A：对应指导书中的 `pDependentSetInFollow`。它记录了非终结符 B 依赖于哪些非终结符 A（即 B 的 FOLLOW 受 A 的 FOLLOW 影响）。虽然在当前的不动点迭代实现中这个字段不直接驱动算法（外层 loop 已经保证了传递），但它保留了依赖关系，便于调试和可视化。

**Q3：SLR(1) 和 LR(0) 的区别在代码中如何体现？**

A：LR(0) 只看项集本身——规约项在任何输入下都规约，所以容易冲突。SLR(1) 引入了 FOLLOW 集——规约项 `A → α·` 只在输入符号属于 `FOLLOW(A)` 时才规约。代码中 `build_lr_parse_table` 的第 139 行：`for t in &follow`——这就是 SLR(1) 的核心：用 FOLLOW 集限制规约的适用范围。

**Q4：ε 产生式在代码中如何表示？**

A：在 `grammar.rs` 第 65-69 行，如果产生式体是 `["ε"]`，会转换为空的 `Vec::new()`。在 `first_of_sequence` 中（第 8-11 行），空序列直接返回 `{ε}`。在 `is_epsilon()` 方法中，`body.is_empty()` 返回 `true` 即表示 ε 产生式。

**Q5：为什么 `Grammar::new` 中要自动创建增广文法？**

A：增广文法是 LR 分析的必需步骤。增广开始符号 `S'` 有且仅有一条产生式 `S' → S`，这保证了：
