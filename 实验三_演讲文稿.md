## 第一部分：实验目标与指导书核心思想（约 2 分钟）

### 1.1 实验指导书要求回顾

指导书第 9-15 页详细描述了词法分析器构造工具（简称"工具"）的实现方法，核心思想是**语法制导的翻译（Syntax-Directed Translation, SDT）**：

- **输入**：一个用正则语言写出的程序，该程序描述了某门高级程序语言（目标语言）的词法。
- **输出**：目标语言词法的 DFA（确定有限状态自动机）。

### 1.2 为什么要用正则语言描述正则语言？

指导书中提出了一个精妙的自举思想：

> 工具的输入是用正则语言写出的程序 → 工具首先要基于正则语言的词法将其切分成词 → 然后基于正则语言的语法得出语法分析树 → 每一步规约的物理含义体现在产生式的 SDD 上。

因此，工具内部包含了两层解析：

1. **正则语言自身的词法分析器**：将输入文本切分为 token（如 `character`、`letter`、`@reserved`、`→` 等）
2. **正则语言自身的语法分析器**：解析语句 `S → id → E crlf`，在规约过程中通过 SDT 构建 NFA，最终转换为 DFA

### 1.3 指导书表 4.6 与表 4.7

指导书的表 4.6 给出了正则语言自身的词法描述（8 个语句），表 4.7 给出了正则语言的文法描述（15 个产生式）。我们项目中对这个格式做了适配——用更直观的 `->` 替代 `→`，但在核心逻辑上完全遵循指导书的设计。

### 2.1 定义解析（parse_regex_definitions 函数）

这是整个工具的入口解析器。指导书中的正则语言程序由若干条语句组成，每条语句格式为 `[@]name → regex_pattern`，其中：
- 带 `@` 前缀的名字表示一个词类（token category）
- 不带 `@` 的名字是辅助定义（如 `character`、`letter`）
- 最后一条语句（通常叫 `lexeme`）是最终结果，合并所有 token

> **⚠️ 两个常量的区别**：`mod.rs` 中定义了两个正则语言程序。**只有 `TINY_LEX_FULL_DEF` 描述了 TINY 语言的词法**——它定义了关键字、标识符、整数常量、运算符、界符、注释等 18 类 token，并最终被 `build_tiny_lexer_dfa()` 传入解析器构建 DFA。而 `TINY_LEX_DEF` 描述的是正则语言自身的词法（仅用于自举演示）。
> 
> | 常量 | 描述的是什么语言？ | 能否描述 TINY 词法？ | 用途 |
> |------|-------------------|--------------------|------|
> | `TINY_LEX_DEF` | 正则语言自身的词法（自举演示） | ❌ 不能 | 在 `run_lab3_test()` 打印，展示 "用正则语言描述词法" 的格式 |
> | `TINY_LEX_FULL_DEF` | **TINY 语言**的完整词法 | ✅ **能** | 传入 `build_tiny_lexer_dfa()`，构建 TINY 词法 DFA |

**`TINY_LEX_DEF`**（正则语言自举描述，`mod.rs` 第 19-28 行）：

```rust
pub const TINY_LEX_DEF: &str = r#"
character -> '\0'..'\127'   // 所有可见与不可见字符的总范围
letter -> 'a'..'z' | 'A'..'Z' // 字母集合，供标识符和名字引用使用
@reserved -> '(' | ')' | '|' | '·' | '*' | '+' | '?' | '→' | '@' | '$' | '-' | '~' // 正则语言自身的保留符号
@id -> letter+                // 一个或多个字母组成名字
@cc -> ''' character '''      // 字符常量，形如 'a'
@space -> ' '+                // 一个或多个空格
@crlf -> ('\r' · '\n')+      // Windows 风格换行
lexeme -> reserved | id | cc | space | crlf // 最终词类并集
"#;
```

**讲解要点**：这个程序描述的是正则语言自身的 token 类别（`@reserved` 是正则运算符、`@id` 是名字引用、`@cc` 是字符常量），用于展示"用正则语言描述另一门语言词法"的自举思想。

**`TINY_LEX_FULL_DEF`**（TINY 语言完整词法定义，`mod.rs` 第 32-54 行）：

指导书要求我们以 TINY 语言的词法来验证程序代码的正确性。我在 `TINY_LEX_FULL_DEF` 中定义了更完整的 token 类别：

```rust
pub const TINY_LEX_FULL_DEF: &str = r#"
digit           -> '0'..'9'              // 只匹配单个数字字符，是整数常量的基础
letter          -> 'a'..'z' | 'A'..'Z'    // 只匹配单个字母，是标识符和关键字的基础
letter_or_digit -> letter | digit         // 标识符后缀允许字母或数字，便于连续组成名字

@keyword    -> 'i' 'f' | 't' 'h' 'e' 'n' | 'e' 'l' 's' 'e' | 'e' 'n' 'd' // 关键字逐字母精确匹配，避免被识别成 ID
         | 'r' 'e' 'p' 'e' 'a' 't' | 'u' 'n' 't' 'i' 'l'
         | 'r' 'e' 'a' 'd' | 'w' 'r' 'i' 't' 'e'

@numeric_op    -> '+' | '-' | '*' | '/'   // 算术运算符，单字符 token
@compare_op    -> '<' | '='               // 比较运算符，后续可扩展到 <= 等形式
@logic_op      -> '(' | ')' | ';'         // 逻辑界符和分号，控制语句结构用
@assign        -> ':' '='                  // 赋值符号，必须整体识别为 :=，不能拆开

@id            -> letter letter_or_digit* // 标识符：第 1 个字符必须是字母，后面可重复出现字母或数字

@integer_const -> digit+                   // 整数常量，至少 1 位数字
@note          -> '{' character* '}'       // 注释内容，花括号内允许任意字符
@space         -> ' ' | '\t' | '\r' | '\n' // 空白符集合，扫描时会被过滤掉

lexeme -> keyword | numeric_op | compare_op | logic_op | assign // 第一层先并上固定符号类
    | id | integer_const | note | space                     // 第二层再并上标识符、常量、注释和空白
"#;
```

**讲解要点**：
- `@keyword` 使用隐式连接运算（字符之间直接相邻），将关键字逐字母拼出。这是一个巧妙的设计——不使用正则语言的闭包运算来匹配关键字，而是逐字精确匹配，避免了标识符与关键字混淆的问题。
- `@id -> letter letter_or_digit*` 定义标识符：以字母开头，后跟零个或多个字母或数字
- `@integer_const -> digit+` 使用正闭包 `+` 定义整数常量
- `@note -> '{' character* '}'` 使用闭包 `*` 定义注释（花括号内可包含任意字符）
- `@space` 定义了空白字符集合（空格、制表符、回车、换行）
- 最后的 `lexeme -> ...` 将所有 token 类别做并运算

**解析函数** `parse_regex_definitions`（`regex_lang.rs` 第 21-61 行）：

```rust
pub fn parse_regex_definitions(text: &str) -> Vec<RegexDef> {
    let mut defs = Vec::new(); // 最终返回的定义列表
    let mut current: Option<RegexDef> = None; // 当前正在拼接的定义，处理多行续写

    for line in text.lines() {
        let original = line.trim(); // 去掉左右空白，避免格式影响解析
        if original.is_empty() { continue; } // 空行直接跳过
        let line = skip_leading_number(original); // 去掉行号前缀，方便处理文稿式文本

        // 续行：以 | 开头，说明这一行仍然属于上一条正则定义
        if line.starts_with('|') {
            if let Some(ref mut def) = current {
                let continuation = line[1..].trim(); // 去掉 | 后的空白，取真正的续行内容
                def.pattern.push(' ');
                def.pattern.push('|');
                def.pattern.push(' ');
                def.pattern.push_str(continuation);
            }
            continue;
        }
        // 当前行不是续行时，先把上一条定义保存到结果列表
        if let Some(def) = current.take() {
            defs.push(def);
        }
        // 再解析新的一条定义: [@]name -> regex_pattern
        if let Some(def) = parse_one_definition(line) {
            current = Some(def);
        }
    }
    if let Some(def) = current.take() { defs.push(def); }
    defs
}
```

**讲解要点**：
- 支持多行定义续行（以 `|` 开头拼接）
- `@` 前缀检测决定 `is_token` 属性
- 返回 `Vec<RegexDef>` 列表，每条含 `name`、`pattern`、`is_token`

### 2.2 正则表达式 AST 与递归下降解析

指导书的表 4.9（SDD）中列出了 10 种正则运算：并 `|`、连接（显式 `·` 和隐式）、闭包 `*`、正闭包 `+`、可选 `?`、括号 `()`、范围运算 `~`、差运算 `-`。我们的解析器完整支持所有这些运算。

**AST 节点定义**（`regex_lang.rs` 第 96-109 行）：

```rust
enum RegexAST {
    CharSet(i32, Option<char>),  // 字符集 ID 及可选关联字符
    CharLiteral(char),           // 字符字面量
    Name(String),                // 引用其他命名正则
    Union(Box<RegexAST>, Box<RegexAST>),
    Concat(Box<RegexAST>, Box<RegexAST>),
    Star(Box<RegexAST>),
    Plus(Box<RegexAST>),
    Option_(Box<RegexAST>),
    Range(char, char),           // 字符范围运算: 'a'~'z' 或 'a'..'z'
    Difference(Box<RegexAST>, Box<RegexAST>),
}
```

**递归下降解析器结构**（优先级从低到高）：

```
parse_union       → 处理 | (并运算，最低优先级)
  parse_concat    → 处理隐式连接和显式 ·
    parse_range_diff → 处理范围 .. 、 ~ 、差运算 -
      parse_unary → 处理一元后缀 *, +, ?
        parse_atom → 处理括号、字符字面量、名字引用
```

**核心解析代码——并运算**（第 276-285 行）：

```rust
fn parse_union(tokens: &[RegexTok], pos: usize) -> (RegexAST, usize) {
    let (mut left, mut pos) = parse_concat(tokens, pos); // 先解析更高优先级的连接表达式
    while pos < tokens.len() && tokens[pos] == RegexTok::Pipe {
        pos += 1; // 跳过 |
        let (right, new_pos) = parse_concat(tokens, pos); // 右侧仍然按连接优先级解析
        left = RegexAST::Union(Box::new(left), Box::new(right)); // 构造并运算 AST
        pos = new_pos; // 继续向后读取同优先级的并运算链
    }
    (left, pos)
}
```

**讲解要点**：这是一个标准的递归下降解析器模式——先解析更高优先级的子表达式作为左操作数，然后循环处理同优先级的二元运算符（此处为 `|`），构建 Union 节点。

**核心解析代码——闭包 / 正闭包 / 可选**（第 376-396 行）：

```rust
fn parse_unary(tokens: &[RegexTok], pos: usize) -> (RegexAST, usize) {
    let (mut ast, mut pos) = parse_atom(tokens, pos); // 先拿到原子项
    while pos < tokens.len() {
        match tokens[pos] {
            RegexTok::Star => {
                ast = RegexAST::Star(Box::new(ast)); // * 表示闭包
                pos += 1;
            }
            RegexTok::Plus => {
                ast = RegexAST::Plus(Box::new(ast)); // + 表示正闭包
                pos += 1;
            }
            RegexTok::Question => {
                ast = RegexAST::Option_(Box::new(ast)); // ? 表示可选
                pos += 1;
            }
            _ => break,
        }
    }
    (ast, pos)
}
```

**讲解要点**：一元后缀运算符 `*`、`+`、`?` 紧跟在原子之后，优先级高于所有二元运算。循环处理允许连续多个一元运算符（如 `a*+` 虽然实际很少这样写）。

### 2.3 AST 求值与 NFA 构建（eval_ast + eval_ast_inner）

这是 SDD 的核心——将 AST 节点转换为 NFA。对应指导书表 4.9 中的 `generateBasicNFA`、`union`、`product`、`closure`、`plusClosure`、`zeroOrOne` 等函数。

```rust
fn eval_ast_inner(ast: &RegexAST, env: &HashMap<String, Graph>) -> Graph {
    match ast {
        RegexAST::CharSet(id, _) => {
            let segs = segments_of(*id); // 查看该字符集是否退化成单字符
            if segs.len() == 1 && segs[0].fromChar == segs[0].toChar {
                generateBasicNFA("CHAR", *id, None)  // 单字符直接建 NFA
            } else {
                generateBasicNFA("CHARSET", *id, None)  // 字符集按集合边建 NFA
            }
        }
        RegexAST::Name(name) => {
            env.get(name).cloned().unwrap_or(Graph::new(0))  // 递归引用已定义的名字
        }
        RegexAST::Union(l, r) => {
            let l_nfa = eval_ast_inner(l, env);
            let r_nfa = eval_ast_inner(r, env);
            union(l_nfa, r_nfa)          // 组合成并运算 NFA
        }
        RegexAST::Concat(l, r) => {
            let l_nfa = eval_ast_inner(l, env);
            let r_nfa = eval_ast_inner(r, env);
            product(l_nfa, r_nfa)        // 组合成连接运算 NFA
        }
        RegexAST::Star(inner) => {
            let nfa = eval_ast_inner(inner, env);
            closure(nfa)                 // 0 次或多次
        }
        RegexAST::Plus(inner) => {
            let nfa = eval_ast_inner(inner, env);
            plusClosure(nfa)             // 1 次或多次
        }
        RegexAST::Option_(inner) => {
            let nfa = eval_ast_inner(inner, env);
            zeroOrOne(nfa)               // 0 次或 1 次
        }
        RegexAST::Range(from_c, to_c) => {
            let id = range(*from_c, *to_c);  // 先把范围登记成字符集 ID
            generateBasicNFA("CHARSET", id, None)
        }
        RegexAST::Difference(l, r) => {
            // 差运算：例如 letter - 'i' 或 letter - digit
            let l_id = extract_charset_id(l, env);
            let r_id = extract_charset_id(r, env);
            let r_char = extract_char(r);
            let result_id = match (l_id, r_id, r_char) {
                (Some(lid), _, Some(c)) => difference_charset_char(lid, c),
                (Some(lid), Some(rid), _) => difference_charsets(lid, rid),
                _ => 0,
            };
            generateBasicNFA("CHARSET", result_id, None) // 差集结果继续作为字符集建模
        }
    }
}
```

**讲解要点**：
- `env: HashMap<String, Graph>` 是名字到 NFA 的映射表，方便当前定义引用前面已经定义过的名字。对应指导书中的 `map`（set）和 `unmap`（get）函数
- `eval_ast_inner` 递归遍历 AST，对每个节点调用对应的 NFA 构建函数
- 字符集差运算 `Difference` 支持两种模式：字符集-字符（如 `letter - 'i'`）和字符集-字符集（如 `letter - digit`）
- `is_token` 标记的词类会在 `eval_ast` 中设置结束状态的 `LexemeCategory`（第 444-451 行）

### 2.4 eval_ast — 把 AST 真正落成 NFA

如果只讲 `eval_ast_inner`，还差最后一步：它负责根据 AST 递归生成 NFA，但还没把“这是不是一个词类”这件事标出来，所以外层再包一层 `eval_ast`。

```rust
fn eval_ast(
    ast: &RegexAST,
    env: &HashMap<String, Graph>,
    is_token: bool,
    token_name: &str,
) -> Graph {
    let mut nfa = eval_ast_inner(ast, env); // 先按 AST 的结构生成基础 NFA

    if is_token {
        let cat = name_to_category(token_name); // 根据名字确定最终词类
        for state in &mut nfa.pStateTable {
            if state.StateType == "MATCH" {
                state.LexemeCategory = Some(cat.clone()); // 接受态挂上词类，供扫描阶段识别
            }
        }
    }

    nfa // 返回已经带有词类信息的 NFA
}
```

**讲解要点**：
- `ast` 是解析后的正则表达式结构，表示当前这一条定义的语义内容。
- `env` 保存的是名字到 NFA 的映射，方便当前定义引用前面已经定义过的名字。
- `is_token` 用来判断这是不是词类定义；只有词类才需要把接受态标上类别。
- `token_name` 是当前名字，比如 `keyword`、`id`、`integer_const`，它会被转换成 `LexemeCategory`。
- `eval_ast_inner` 负责“造图”，`eval_ast` 负责“造图之后再补词类信息”。
- 最终返回的是一个能继续做 DFA 转换的 NFA，而不是 AST 本身。

一句话概括就是：`eval_ast_inner` 负责结构，`eval_ast` 负责属性。

### 3.2 build_nfa_from_defs — 从定义列表生成 NFA

这个函数在 `src/lab3/regex_lang.rs` 中，是实验三里“真正把正则写法变成自动机”的部分。

```rust
pub fn build_nfa_from_defs(defs: &[RegexDef]) -> (Graph, HashMap<String, LexemeCategory>) {
    let mut env: HashMap<String, Graph> = HashMap::new(); // 保存名字到 NFA 的映射，供后续定义引用
    let mut token_map: HashMap<String, LexemeCategory> = HashMap::new(); // 保存 token 名到词类的映射

    for def in defs {
        let ast = parse_regex_pattern(&def.pattern); // 把字符串正则解析成 AST
        let nfa = eval_ast(&ast, &env, def.is_token, &def.name); // 依据 AST 和已有环境生成 NFA
        env.insert(def.name.clone(), nfa); // 当前名字的 NFA 先放入环境，供后面的定义复用

        if def.is_token {
            let cat = name_to_category(&def.name); // 把名字转换成实际词类
            token_map.insert(def.name.clone(), cat); // 记录这个名字对应哪个 token 类别
        }
    }

    let final_nfa = if let Some(last) = defs.last() {
        env.get(&last.name).cloned().unwrap_or_else(|| { // 最后一条通常是 lexeme；若未命中则合并所有 token
            merge_all_token_nfas(&env, &defs, &token_map)
        })
    } else {
        Graph::new(0) // 没有定义时返回空图，避免后续流程崩溃
    };

    (final_nfa, token_map)
}
```

**讲解要点**：
- `env` 保存的是“名字 → NFA”的映射，后面的定义可以引用前面的定义
- 每条定义先解析成 AST，再通过 `eval_ast` 生成 NFA
- `is_token` 为真时，会把这个定义记成真正的词类
- 最后一条 `lexeme` 通常是总入口，它把所有 token 组合成最终 NFA

随后程序调用 `build_tiny_lexer_dfa()`，从 `TINY_LEX_FULL_DEF` 出发，经解析 → AST 求值 → NFA → DFA → 最小化，输出最终状态数：
