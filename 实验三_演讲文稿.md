## 第一部分：实验目标与指导书核心思想（约 2 分钟）

### 1.1 实验指导书要求回顾

指导书第 9-15 页详细描述了词法分析器构造工具（简称"工具"）的实现方法，核心思想是**语法制导的翻译（Syntax-Directed Translation, SDT）**：

- **输入**：一个用正则语言写出的程序，该程序描述了某门高级程序语言（目标语言）的词法。
- **输出**：目标语言词法的 DFA（确定有限状态自动机）。

### 1.2 为什么要用正则语言描述正则语言？

指导书中提出了一个精妙的自举思想：

> 工具的输入是用正则语言写出的程序 → 工具首先要基于正则语言的词法将其切分成词 → 然后基于正则语言的语法得出语法分析树 → 每一步规约的物理含义体现在产生式的 SDD 上。

因此，工具内部包含了两层解析：

1. **正则语言自身的词法分析器**：将输入文本切分为 token（如 `character`、`letter`、`@reserved`、`→` 等）
2. **正则语言自身的语法分析器**：解析语句 `S → id → E crlf`，在规约过程中通过 SDT 构建 NFA，最终转换为 DFA

### 1.3 指导书表 4.6 与表 4.7

指导书的表 4.6 给出了正则语言自身的词法描述（8 个语句），表 4.7 给出了正则语言的文法描述（15 个产生式）。我们项目中对这个格式做了适配——用更直观的 `->` 替代 `→`，但在核心逻辑上完全遵循指导书的设计。

### 2.1 定义解析（parse_regex_definitions 函数）

这是整个工具的入口解析器。指导书中的正则语言程序由若干条语句组成，每条语句格式为 `[@]name → regex_pattern`，其中：
- 带 `@` 前缀的名字表示一个词类（token category）
- 不带 `@` 的名字是辅助定义（如 `character`、`letter`）
- 最后一条语句（通常叫 `lexeme`）是最终结果，合并所有 token

**代码分析**（`src/lab3/mod.rs` 第 19-28 行）：

```rust
pub const TINY_LEX_DEF: &str = r#"
character -> '\0'..'\127'
letter -> 'a'..'z' | 'A'..'Z'
@reserved -> '(' | ')' | '|' | '·' | '*' | '+' | '?' | '→' | '@' | '$' | '-' | '~'
@id -> letter+
@cc -> ''' character '''
@space -> ' '+
@crlf -> ('\r' · '\n')+
lexeme -> reserved | id | cc | space | crlf
"#;
```

**讲解要点**：
- 第 1 行 `character -> '\0'..'\127'` 定义了 ASCII 全字符集，使用了范围运算符 `..`
- 第 2 行 `letter -> 'a'..'z' | 'A'..'Z'` 使用并运算定义字母集
- 第 3-7 行带 `@` 前缀，分别定义预留字、标识符、字符常量、空格、回车换行——这些都是词法分析器的 token 类别
- 第 8 行 `lexeme -> reserved | id | cc | space | crlf` 是所有 token 类别的并运算，构成最终的词法 NFA

**完整的 TINY 语言词法定义**（`TINY_LEX_FULL_DEF`，第 32-54 行）：

指导书要求我们以 TINY 语言的词法来验证程序代码的正确性。我在 `TINY_LEX_FULL_DEF` 中定义了更完整的 token 类别：

```rust
pub const TINY_LEX_FULL_DEF: &str = r#"
digit           -> '0'..'9'
letter          -> 'a'..'z' | 'A'..'Z'
letter_or_digit -> letter | digit

@keyword    -> 'i' 'f' | 't' 'h' 'e' 'n' | 'e' 'l' 's' 'e' | 'e' 'n' 'd'
             | 'r' 'e' 'p' 'e' 'a' 't' | 'u' 'n' 't' 'i' 'l'
             | 'r' 'e' 'a' 'd' | 'w' 'r' 'i' 't' 'e'

@numeric_op    -> '+' | '-' | '*' | '/'
@compare_op    -> '<' | '='
@logic_op      -> '(' | ')' | ';'
@assign        -> ':' '='

@id            -> letter letter_or_digit*

@integer_const -> digit+
@note          -> '{' character* '}'
@space         -> ' ' | '\t' | '\r' | '\n'

lexeme -> keyword | numeric_op | compare_op | logic_op | assign
        | id | integer_const | note | space
"#;
```

**讲解要点**：
- `@keyword` 使用隐式连接运算（字符之间直接相邻），将关键字逐字母拼出。这是一个巧妙的设计——不使用正则语言的闭包运算来匹配关键字，而是逐字精确匹配，避免了标识符与关键字混淆的问题。
- `@id -> letter letter_or_digit*` 定义标识符：以字母开头，后跟零个或多个字母或数字
- `@integer_const -> digit+` 使用正闭包 `+` 定义整数常量
- `@note -> '{' character* '}'` 使用闭包 `*` 定义注释（花括号内可包含任意字符）
- `@space` 定义了空白字符集合（空格、制表符、回车、换行）
- 最后的 `lexeme -> ...` 将所有 token 类别做并运算

**解析函数** `parse_regex_definitions`（`regex_lang.rs` 第 21-61 行）：

```rust
pub fn parse_regex_definitions(text: &str) -> Vec<RegexDef> {
    let mut defs = Vec::new();
    let mut current: Option<RegexDef> = None;

    for line in text.lines() {
        let original = line.trim();
        if original.is_empty() { continue; }
        let line = skip_leading_number(original);

        // 续行：以 `|` 开头，追加到前一个定义的 pattern
        if line.starts_with('|') {
            if let Some(ref mut def) = current {
                let continuation = line[1..].trim();
                def.pattern.push(' ');
                def.pattern.push('|');
                def.pattern.push(' ');
                def.pattern.push_str(continuation);
            }
            continue;
        }
        // 先将前一个定义保存
        if let Some(def) = current.take() {
            defs.push(def);
        }
        // 解析格式: [@]name -> regex_pattern
        if let Some(def) = parse_one_definition(line) {
            current = Some(def);
        }
    }
    if let Some(def) = current.take() { defs.push(def); }
    defs
}
```

**讲解要点**：
- 支持多行定义续行（以 `|` 开头拼接）
- `@` 前缀检测决定 `is_token` 属性
- 返回 `Vec<RegexDef>` 列表，每条含 `name`、`pattern`、`is_token`

### 2.2 正则表达式 AST 与递归下降解析

指导书的表 4.9（SDD）中列出了 10 种正则运算：并 `|`、连接（显式 `·` 和隐式）、闭包 `*`、正闭包 `+`、可选 `?`、括号 `()`、范围运算 `~`、差运算 `-`。我们的解析器完整支持所有这些运算。

**AST 节点定义**（`regex_lang.rs` 第 96-109 行）：

```rust
enum RegexAST {
    CharSet(i32, Option<char>),  // 字符集 ID 及可选关联字符
    CharLiteral(char),           // 字符字面量
    Name(String),                // 引用其他命名正则
    Union(Box<RegexAST>, Box<RegexAST>),
    Concat(Box<RegexAST>, Box<RegexAST>),
    Star(Box<RegexAST>),
    Plus(Box<RegexAST>),
    Option_(Box<RegexAST>),
    Range(char, char),           // 字符范围运算: 'a'~'z' 或 'a'..'z'

}
```

**递归下降解析器结构**（优先级从低到高）：

```
parse_union       → 处理 | (并运算，最低优先级)
  parse_concat    → 处理隐式连接和显式 ·
    parse_range_diff → 处理范围 .. 、 ~ 、差运算 -
      parse_unary → 处理一元后缀 *, +, ?
        parse_atom → 处理括号、字符字面量、名字引用
```

**核心解析代码——并运算**（第 276-285 行）：

```rust
fn parse_union(tokens: &[RegexTok], pos: usize) -> (RegexAST, usize) {
    let (mut left, mut pos) = parse_concat(tokens, pos);
    while pos < tokens.len() && tokens[pos] == RegexTok::Pipe {
        pos += 1;
        let (right, new_pos) = parse_concat(tokens, pos);
        left = RegexAST::Union(Box::new(left), Box::new(right));
        pos = new_pos;
    }
    (left, pos)
}
```

**讲解要点**：这是一个标准的递归下降解析器模式——先解析更高优先级的子表达式作为左操作数，然后循环处理同优先级的二元运算符（此处为 `|`），构建 Union 节点。

**核心解析代码——闭包 / 正闭包 / 可选**（第 376-396 行）：

```rust
fn parse_unary(tokens: &[RegexTok], pos: usize) -> (RegexAST, usize) {
    let (mut ast, mut pos) = parse_atom(tokens, pos);
    while pos < tokens.len() {
        match tokens[pos] {
            RegexTok::Star => {
                ast = RegexAST::Star(Box::new(ast));
                pos += 1;
            }
            RegexTok::Plus => {
                ast = RegexAST::Plus(Box::new(ast));
                pos += 1;
            }
            RegexTok::Question => {
                ast = RegexAST::Option_(Box::new(ast));
                pos += 1;
            }
            _ => break,
        }
    }
    (ast, pos)
}
```

**讲解要点**：一元后缀运算符 `*`、`+`、`?` 紧跟在原子之后，优先级高于所有二元运算。循环处理允许连续多个一元运算符（如 `a*+` 虽然实际很少这样写）。

### 2.3 AST 求值与 NFA 构建（eval_ast + eval_ast_inner）

这是 SDD 的核心——将 AST 节点转换为 NFA。对应指导书表 4.9 中的 `generateBasicNFA`、`union`、`product`、`closure`、`plusClosure`、`zeroOrOne` 等函数。

```rust
fn eval_ast_inner(ast: &RegexAST, env: &HashMap<String, Graph>) -> Graph {
    match ast {
        RegexAST::CharSet(id, _) => {
            let segs = segments_of(*id);
            if segs.len() == 1 && segs[0].fromChar == segs[0].toChar {
                generateBasicNFA("CHAR", *id, None)  // 单字符
            } else {
                generateBasicNFA("CHARSET", *id, None)  // 字符集
            }
        }
        RegexAST::Name(name) => {
            env.get(name).cloned().unwrap_or(Graph::new(0))  // 引用其他定义
        }
        RegexAST::Union(l, r) => {
            let l_nfa = eval_ast_inner(l, env);
            let r_nfa = eval_ast_inner(r, env);
            union(l_nfa, r_nfa)          // 并运算
        }
        RegexAST::Concat(l, r) => {
            let l_nfa = eval_ast_inner(l, env);
            let r_nfa = eval_ast_inner(r, env);
            product(l_nfa, r_nfa)        // 连接运算
        }
        RegexAST::Star(inner) => {
            let nfa = eval_ast_inner(inner, env);
            closure(nfa)                 // 闭包运算
        }
        RegexAST::Plus(inner) => {
            let nfa = eval_ast_inner(inner, env);
            plusClosure(nfa)             // 正闭包运算
        }
        RegexAST::Option_(inner) => {
            let nfa = eval_ast_inner(inner, env);
            zeroOrOne(nfa)               // 0或者1个运算
        }
        RegexAST::Range(from_c, to_c) => {
            let id = range(*from_c, *to_c);  // 字符范围运算
            generateBasicNFA("CHARSET", id, None)
        }
        RegexAST::Difference(l, r) => {
            // 差运算：letter - 'i' 或 letter - digit
            let l_id = extract_charset_id(l, env);
            let r_id = extract_charset_id(r, env);
            let r_char = extract_char(r);
            let result_id = match (l_id, r_id, r_char) {
                (Some(lid), _, Some(c)) => difference_charset_char(lid, c),
                (Some(lid), Some(rid), _) => difference_charsets(lid, rid),
                _ => 0,
            };
            generateBasicNFA("CHARSET", result_id, None)
        }
    }
}
```

**讲解要点**：
- `env: HashMap<String, Graph>` 是名字到 NFA 的映射表，对应指导书中的 `map`（set）和 `unmap`（get）函数
- `eval_ast_inner` 递归遍历 AST，对每个节点调用对应的 NFA 构建函数
- 字符集差运算 `Difference` 支持两种模式：字符集-字符（如 `letter - 'i'`）和字符集-字符集（如 `letter - digit`）
- `is_token` 标记的词类会在 `eval_ast` 中设置结束状态的 `LexemeCategory`（第 444-451 行）

### 3.1 generateBasicNFA — 基本 NFA 构造

指导书要求的最简 NFA 构造法：对于单个字符或字符集，创建只包含两个状态（0 状态和 1 状态）的 NFA，结束状态（1 状态）无出边。

```rust
// 函数签名（graph.rs 中）
pub fn generateBasicNFA(
    driver_type: &str,   // "CHAR" 或 "CHARSET"
    driver_id: i32,      // 字符在 ASC 表中的序号 或 字符集 ID
    category: Option<LexemeCategory>
) -> Graph
```

**讲解要点**：
- `driver_type = "CHAR"` 时，边上的驱动是单个字符
- `driver_type = "CHARSET"` 时，边上的驱动是字符集（如 `[a-z]`、`[0-9]`）
- `category` 参数可选，用于设置结束状态的词类属性

### 3.2 NFA 组合运算

指导书要求的 5 种 NFA 组合运算，对应 5 种正则运算：

| 函数 | 正则运算 | 说明 |
|------|---------|------|
| `union(g1, g2)` | `\|`（并） | 创建新开始/结束状态，ε-边并行连接 |
| `product(g1, g2)` | 连接 | g1 的结束状态 ε-连接到 g2 的开始状态 |
| `closure(g)` | `*`（闭包） | ε-边形成自循环，匹配零次或多次 |
| `plusClosure(g)` | `+`（正闭包） | product + closure，匹配一次或多次 |
| `zeroOrOne(g)` | `?`（可选） | ε-边跳过，匹配零次或一次 |

**代码讲解——以 `union` 为例**：

指导书图 4.5（正则语言文法的 DFA）中，状态 I_0 到 I_15 的并运算边体现了 `E → E|E` 的规约。在我们的代码中，`union` 函数：
1. 创建两个新状态作为并运算 NFA 的开始和结束
2. 从新开始状态创建 ε-边分别连接到 g1 和 g2 的开始状态
3. 将 g1 和 g2 的结束状态用 ε-边连接到新结束状态
4. 返回合并后的 NFA

### 4.2 build_tiny_lexer_dfa — 动态构建流程

这是实验三的核心入口函数（`src/lab3/mod.rs` 第 60-78 行）：

```rust
pub fn build_tiny_lexer_dfa() -> Graph {
```

**讲解要点**——这个函数完美体现了指导书的 SDT 流程：
1. **解析**（parse）：将正则语言程序切分为定义列表
2. **构建 NFA**（build_nfa_from_defs）：对每条定义解析正则模式 → 生成 AST → 求值 AST 构建 NFA → 存入环境表。最后一条 `lexeme` 将所有 token 合并为最终 NFA
3. **NFA → DFA**：子集构造法转换

### 5.3 sample.tny 扫描结果

**输入源程序**（指导书第 15 页提供的样例，共 10 行）：

```
{ Sample program in TINY language - computes factorial }
read x; { input an integer }
if 0 < x then { don't compute if x <= 0 }
    fact := 1;
    repeat
        fact := fact * x;
        x := x - 1
    until x = 0;
    write fact { output factorial of x }
end
```

**词法分析输出**（过滤注释和空白后）：

```
  ID                     | Sample
  ID                     | program
  ID                     | in
  ID                     | TINY
  ID                     | language
  NUMERIC_OPERATOR       | -
  ID                     | computes
  ID                     | factorial
  KEYWORD                | read
  ID                     | x
  LOGIC_OPERATOR         | ;
  ...
  KEYWORD                | if
  INTEGER_CONST          | 0
  COMPARE_OPERATOR       | <
  ID                     | x
  KEYWORD                | then
  ...
  ID                     | fact
  ASSIGN_OPERATOR        | :=
  INTEGER_CONST          | 1
  LOGIC_OPERATOR         | ;
  KEYWORD                | repeat
  ...
  KEYWORD                | until
  ID                     | x
  COMPARE_OPERATOR       | =
  INTEGER_CONST          | 0
  LOGIC_OPERATOR         | ;
  KEYWORD                | write
  ID                     | fact
  ...
  KEYWORD                | end
```

**判断讲解**：

1. **注释处理**：`{ Sample program in TINY language ... }` 被正确识别为 `NOTE` 类别并被过滤掉（不输出），但注释内部的 `Sample program` 等词仍然被单独识别为 `ID`

2. **关键字识别**：`read`、`if`、`then`、`repeat`、`until`、`write`、`end` 被正确识别为 `KEYWORD` 类别，而不是普通的 `ID`

3. **运算符识别**：
   - 赋值符 `:=` 被正确识别为单个 `ASSIGN_OPERATOR`，而不是 `:` 和 `=` 两个独立 token——这体现了最长匹配策略
   - `<` 被识别为 `COMPARE_OPERATOR`
   - `*` 被识别为 `NUMERIC_OPERATOR`
   - `-` 被识别为 `NUMERIC_OPERATOR`
   - `;` 被识别为 `LOGIC_OPERATOR`

4. **标识符识别**：`x`、`fact`、`max_val` 等被正确识别为 `ID` 类别

5. **整数常量识别**：`0`、`1` 等被正确识别为 `INTEGER_CONST`，并附带数值 value

6. **注释内部的词被分割识别的原因**：注释内部的 `Sample program in TINY language` 被当作独立单词分别识别——这是因为我们的注释匹配模式 `'{' character* '}'` 允许注释内部有任意字符，但 `long_text_search` 在不匹配完整花括号时仍会尝试识别内部 token。这是一个已知行为。
