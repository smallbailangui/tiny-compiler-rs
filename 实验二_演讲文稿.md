## 第一部分：数据结构设计（约 3 分钟）

> 对应指导书第 5-7 页：GrammarSymbol、Production、LR0Item、ItemSet、DFA、语法分析表

### 1.1 文法符号 `GrammarSymbol`

指导书定义了 `GrammarSymbol`（父类）、`TerminalSymbol`（终结符子类）和 `NonTerminalSymbol`（非终结符子类）。Rust 不支持传统继承，我用一个结构体 + 枚举区分：

```rust
// src/lab2/types.rs
pub struct GrammarSymbol {
    pub name: String,                          // 符号名，如 "E", "+", "id"
    pub sym_type: SymbolType,                  // Terminal / NonTerminal / Null
    pub productions: Vec<usize>,               // 非终结符对应的产生式编号
    pub first_set: BTreeSet<String>,           // FIRST 集
    pub follow_set: BTreeSet<String>,          // FOLLOW 集
    pub dependent_in_follow: BTreeSet<String>, // FOLLOW 求解依赖
}
```

**关键设计决策**：
- 用 `BTreeSet<String>` 存储集合，自动去重 + 字典序排列 → 输出美观
- `dependent_in_follow` 对应指导书中的 `pDependentSetInFollow`，记录"谁依赖谁的 FOLLOW"
- 终结符的 `first_set` 初始化为 `{自身}`（如 `+` 的 FIRST 就是 `{+}`），非终结符初始为空

### 1.2 产生式 `Production`

```rust
pub struct Production {
    pub production_id: usize,         // 序号，0 号为增广产生式 S' → S
    pub head: String,                 // 产生式头（左部非终结符名）
    pub body: Vec<String>,            // 产生式体（右部文法符序列）
    pub first_set: BTreeSet<String>,  // 产生式的 FIRST 集
}
```

**要点**：`body` 为空 `Vec` 表示 ε 产生式（如 `S → ε`）。0 号产生式固定为增广产生式——这是 LR 分析的标准预处理。

### 1.3 LR(0) 项目 `LR0Item`

```rust
pub struct LR0Item {
    pub production_id: usize,  // 产生式编号
    pub dot_position: usize,   // 圆点位置，0 ≤ dot ≤ body.len()
}
```

实现了 `Ord`、`Hash`、`Eq` trait，可直接放入 `BTreeSet` 实现自动去重——指导书中使用 `List<LR0Item*>` + 手动判等，我用标准库替代。

### 1.4 项集 `ItemSet`

```rust
pub struct ItemSet {
    pub state_id: usize,
    pub core_items: BTreeSet<LR0Item>,   // 核心项（用于状态判等）
    pub items: BTreeSet<LR0Item>,         // 完整项集 = 核心项 + 闭包
}
```

**巧妙的分离设计**：`core_items` 专门用于判断"后继项集是否为新项集"（指导书第 8 页要求）。两个项集只要核心项集合相同就是同一状态，避免 DFA 膨胀。这与指导书完全对齐。

### 1.5 变迁边、DFA、语法分析表

```rust
pub struct TransitionEdge {
    pub driver_symbol: String,   // 驱动文法符（终结符或非终结符）
```

**语法分析表使用 `BTreeMap` 而非 `List`**：指导书第 7 页明确建议"最好使用字典类型以提升查找性能"。`(状态编号, 符号名)` 作为复合键，O(log n) 查找效率远优于 List 的 O(n)。

## 第二部分：FIRST 和 FOLLOW 求解（约 4 分钟）

> 对应指导书第 7 页任务 1：FIRST 函数求解、FOLLOW 函数求解

### 2.1 FIRST 集求解算法（不动点迭代）

**产生式的 FIRST** — 函数 `first_of_sequence`：

```rust
// src/lab2/first.rs
pub fn first_of_sequence(symbols: &[String], grammar: &Grammar) -> BTreeSet<String> {
    let mut result = BTreeSet::new();
    let mut all_nullable = true;
    for sym in symbols {
        let sym_first = grammar.symbols[sym].first_set.clone();
        result.extend(sym_first.iter().filter(|&s| s != EPSILON).cloned());
        if !sym_first.contains(EPSILON) {
            all_nullable = false;
            break;
        }
    }
    if all_nullable {
        result.insert(EPSILON.to_string());
    }
    result
}
```

**算法解释**：对序列 `X1 X2 ... Xn`，从 X1 开始取 FIRST，如果不含 ε 就停止；如果含 ε 就继续看 X2...依此类推。全部含 ε 则整个序列的 FIRST 也含 ε。这严格符合教材定义。

**非终结符的 FIRST** — 函数 `compute_first_sets`：

```rust
pub fn compute_first_sets(grammar: &mut Grammar) {
    loop {
        let mut changed = false;
        for prod in &grammar.productions {
            let body_first = first_of_sequence(&prod.body, grammar);
            let head_sym = grammar.symbols.get_mut(&prod.head).unwrap();
            let before = head_sym.first_set.len();
            head_sym.first_set.extend(body_first);
            if head_sym.first_set.len() > before {
                changed = true;
            }
        }
        if !changed { break; }  // 不动点到达
    }
}
```

每轮把每个产生式体的 FIRST 并入头部非终结符。当某轮没有任何变化时，算法终止。**天然处理了直接左递归和间接左递归**——因为不动点迭代会一直扩大直到稳定。

### 2.2 FOLLOW 集求解算法

```rust
// src/lab2/follow.rs
pub fn compute_follow_sets(grammar: &mut Grammar) {
    // 规则1: 开始符号的FOLLOW包含 $
    grammar.symbols.get_mut(&grammar.root).unwrap()
        .follow_set.insert(END_MARKER.to_string());

    loop {
        let mut changed = false;
        for prod in &grammar.productions {
            for i in 0..prod.body.len() {
                let b = &prod.body[i];
                if grammar.is_nonterminal(b) {
                    // 规则2: 取 β = body[i+1..] 的 FIRST - {ε}，加入 FOLLOW(B)
                    let beta_first = first_of_sequence(&prod.body[i+1..], grammar);
                    let before = grammar.symbols[b].follow_set.len();
                    grammar.symbols.get_mut(b).unwrap()
                        .follow_set.extend(beta_first.iter().filter(|&s| s != EPSILON).cloned());
                    // 规则3: 若 β 可推导 ε，则 FOLLOW(A) ⊆ FOLLOW(B)
                    if beta_first.contains(EPSILON) {
                        let head = &prod.head;
                        // 记录依赖关系 → 高效传播
                        grammar.symbols.get_mut(b).unwrap()
                            .dependent_in_follow.insert(head.clone());
                    }
                    if grammar.symbols[b].follow_set.len() > before {
                        changed = true;
                    }
                }
            }
        }
        // 传播依赖：用 dependent_in_follow 把 A 的 FOLLOW 传给它依赖的 B
        // ... (传播逻辑)
        if !changed { break; }
    }
}
```

**三条规则**完全按指导书实现。`dependent_in_follow` 是关键优化——当 `A → αB` 且 `β` 可推导 ε 时，只需记录 "B 依赖 A"，后续传播阶段统一传递，避免重复遍历产生式。

### 2.3 运行结果示例 — 算术表达式文法

运行程序后终端输出：

```
FIRST / FOLLOW 集合
---------------------------------
NT        FIRST              FOLLOW
E'        {(, id}            {$}
E         {(, id}            {$, ), +}
T         {(, id}            {$, ), *, +}
F         {(, id}            {$, ), *, +}
```

**可解释的输出判断**：
- E 的 FIRST = `{(, id}`：E→T→F→`(E)|id`，所以开头只有 `(` 和 `id`
- E 的 FOLLOW = `{$, ), +}`：E 是开始符所以有 `$`；F→`(E)` 所以有 `)`；E→`E+T` 所以有 `+`
- T 的 FOLLOW 比 E 多了 `*`：因为 T→`T*F`，`*` 紧跟在 T 之后
- 所有结果与教材标准答案完全一致 ✅

## 第三部分：LR(0) DFA 构建（约 6 分钟，核心内容）

> 对应指导书第 7-8 页任务 2.1-2.3

### 3.1 函数一：`get_closure` — 闭包求解

```rust
// src/lab2/lr0.rs
pub fn get_closure(core_items: &BTreeSet<LR0Item>, grammar: &Grammar)
    -> BTreeSet<LR0Item>
{
    let mut closure = core_items.clone();
    let mut queue: VecDeque<LR0Item> = core_items.iter().cloned().collect();

    while let Some(item) = queue.pop_front() {
        let prod = &grammar.productions[item.production_id];
        // 圆点后面的符号
        if let Some(next_sym) = prod.body.get(item.dot_position) {
            // 如果是非终结符 A，则加入 A 的所有产生式（圆点在位置 0）
            if grammar.is_nonterminal(next_sym) {
                for &pid in &grammar.symbols[next_sym].productions {
                    let new_item = LR0Item { production_id: pid, dot_position: 0 };
                    if closure.insert(new_item.clone()) {
                        queue.push_back(new_item);  // 新项目入队继续传播
                    }
                }
            }
        }
    }
    closure
}
```

**逐行解释**：
1. 用 `VecDeque` 作为工作队列（BFS）——比递归更安全，避免栈溢出
2. 从队列取出项目，检查圆点后面的符号
3. 如果后面是非终结符 A，就把 A 的**所有**产生式（圆点在位置 0）加入闭包
4. `closure.insert()` 返回 `false` 表示已存在，则不再入队 → 天然去重
5. 队列为空时闭包完成

**对应指导书第 7 页**："穷举一个 LR(0) 项集的闭包，包括核心项和非核心项"。

### 3.2 函数二：`exhaust_transitions` — 穷举变迁

```rust
pub fn exhaust_transitions(item_set: &ItemSet, grammar: &Grammar)
    -> BTreeMap<String, BTreeSet<LR0Item>>
{
    let mut groups: BTreeMap<String, BTreeSet<LR0Item>> = BTreeMap::new();

    for item in &item_set.items {
        let prod = &grammar.productions[item.production_id];
        if let Some(driver) = prod.body.get(item.dot_position) {
            // 圆点后移一位，形成新核心项
            let new_item = LR0Item {
                production_id: item.production_id,
                dot_position: item.dot_position + 1,
            };
            // 按驱动符分组
            groups.entry(driver.clone()).or_default().insert(new_item);
        }
    }
    groups
}
```

**算法解释**：遍历项集所有项目 → 取圆点后符号作为驱动符 → 圆点后移一位作为新核心项 → 按驱动符分组。返回 `BTreeMap<驱动符, BTreeSet<新核心项>>`。例如：

- 驱动符 `+` → 核心项 `{E → E+·T}`
- 驱动符 `id` → 核心项 `{F → id·}`
- 驱动符 `(` → 核心项 `{F → (·E)}`

**对应指导书第 7 页**："驱动符的穷举，后继项集中核心项的确定"。

### 3.3 函数三：`build_lr0_dfa` — DFA 构造

```rust
// src/lab2/lr0.rs
pub fn build_lr0_dfa(grammar: &Grammar) -> LR0DFA {
    // 步骤1: 初始项集 I0 = closure({S' → ·S})
    let start_core: BTreeSet<LR0Item> = [LR0Item { production_id: 0, dot_position: 0 }]
        .into_iter().collect();
    let start_closure = get_closure(&start_core, grammar);

    let mut dfa = LR0DFA {
        start_state: 0,
        item_sets: vec![ItemSet {
            state_id: 0,
            core_items: start_core,
            items: start_closure,
        }],
        edges: Vec::new(),
    };

    let mut processed: HashSet<BTreeSet<LR0Item>> = HashSet::new();
    processed.insert(dfa.item_sets[0].core_items.clone());

    let mut i = 0;
    while i < dfa.item_sets.len() {
        let transitions = exhaust_transitions(&dfa.item_sets[i], grammar);

        for (driver, new_core) in transitions {
            let new_closure = get_closure(&new_core, grammar);
            // 判断是否为新状态：看核心项是否已存在
            if let Some(pos) = dfa.item_sets.iter().position(|s| s.core_items == new_core) {
                // 已有状态，只添加边
                dfa.edges.push(TransitionEdge {
                    driver_symbol: driver,
                    from_state: dfa.item_sets[i].state_id,
                    to_state: pos,
                });
            } else {
                // 新状态
                let new_id = dfa.item_sets.len();
                dfa.item_sets.push(ItemSet {
                    state_id: new_id,
                    core_items: new_core.clone(),
                    items: new_closure,
                });
                processed.insert(new_core);
                dfa.edges.push(TransitionEdge {
                    driver_symbol: driver,
                    from_state: dfa.item_sets[i].state_id,
                    to_state: new_id,
                });
            }
        }
        i += 1;
    }
    dfa
}
```

**逐段讲解**：
1. **初始项集**：`{S' → ·S}` 作为核心，求闭包 → 得到 I0
2. **主循环**：对每个已存在的项集，调用 `exhaust_transitions` 穷举变迁
3. **新状态判定**：`s.core_items == new_core` —— 核心项相同即同一状态（指导书第 8 页"后继项集是否为新项集的判断"）
4. **终止**：所有项集都处理完毕，循环结束

### 3.4 DFA 输出讲解 — 算术表达式文法（12 个状态）

终端实际输出（节选）：

### 4.1 函数四：`check_slr1` — SLR(1) 文法判定

```rust
// src/lab2/slr.rs
pub fn check_slr1(dfa: &LR0DFA, grammar: &Grammar) -> (bool, Vec<String>) {
    let mut conflicts = Vec::new();

    for is in &dfa.item_sets {
        let mut shift_terms: BTreeSet<String> = BTreeSet::new();  // 可移入终结符
        let mut reduce_pids: Vec<usize> = Vec::new();              // 可规约产生式

        for item in &is.items {
            let prod = &grammar.productions[item.production_id];
            if let Some(next_sym) = prod.body.get(item.dot_position) {
                if grammar.is_terminal(next_sym) {
                    shift_terms.insert(next_sym.clone());  // 圆点后有终结符 → 移入
                }
            } else if item.production_id != 0 {
                reduce_pids.push(item.production_id);  // 圆点在末尾 → 规约
            }
        }

        // 检测移入-规约冲突
        for &rpid in &reduce_pids {
            let rhead = grammar.productions[rpid].head.clone();
            let follow = grammar.symbols[&rhead].follow_set.clone();
            for t in &shift_terms {
                if follow.contains(t) {
                    conflicts.push(format!("状态 I{}：移入-规约冲突...", is.state_id));
                }
            }
        }

        // 检测规约-规约冲突
        for i in 0..reduce_pids.len() {
            for j in (i+1)..reduce_pids.len() {
                let f1 = &grammar.symbols[&grammar.productions[reduce_pids[i]].head].follow_set;
                let f2 = &grammar.symbols[&grammar.productions[reduce_pids[j]].head].follow_set;
                if !f1.is_disjoint(f2) {
                    conflicts.push(format!("状态 I{}：规约-规约冲突...", is.state_id));
                }
            }
        }
    }
    (conflicts.is_empty(), conflicts)
}
```

**判断逻辑**：
- 对每个状态，收集可移入的终结符和可规约的产生式
- **移入-规约冲突**：规约项 A→α· 的 FOLLOW(A) 与移入终结符有交集
- **规约-规约冲突**：两个规约项的 FOLLOW 集有交集
- 任一冲突 → 非 SLR(1)

### 4.2 函数五：`build_lr_parse_table` — LR 分析表填写

```rust
// src/lab2/slr.rs
pub fn build_lr_parse_table(dfa: &LR0DFA, grammar: &Grammar) -> LRParseTable {
    // 步骤1: 变迁边 → ACTION/GOTO
    for edge in &dfa.edges {
        if grammar.is_terminal(&edge.driver_symbol) {
            // 终结符 → ACTION 表，填 s（移入）
            action.insert((edge.from_state, edge.driver_symbol.clone()),
                ActionCell { action_type: ActionCategory::Shift, id: edge.to_state });
        } else {
            // 非终结符 → GOTO 表
            goto.insert((edge.from_state, edge.driver_symbol.clone()),
                GotoCell { next_state: edge.to_state });
        }
    }

    // 步骤2: 规约项 → ACTION 表
    for is in &dfa.item_sets {
        for item in &is.items {
            let prod = &grammar.productions[item.production_id];
            if item.dot_position >= prod.body.len() {
                if item.production_id == 0 {
                    // 增广产生式 S' → S· → acc
                    action.insert((is.state_id, "$".to_string()),
                        ActionCell { action_type: ActionCategory::Accept, id: 0 });
                } else {
                    // 普通规约 → 对每个 b ∈ FOLLOW(A)，填 r
                    for t in &grammar.symbols[&prod.head].follow_set {
                        action.insert((is.state_id, t.clone()),
                            ActionCell { action_type: ActionCategory::Reduce, id: item.production_id });
                    }
                }
            }
        }
    }
    LRParseTable { action, goto, conflicts }
}
```

**讲解要点**：
- **移入**：`ACTION[from, driver] = s(to)`，如 `ACTION[0, id] = s5`
- **规约**：`ACTION[state, follow_symbol] = r(production_id)`
- **接受**：`ACTION[state, $] = acc`（仅当 `S' → S·` 时）
- **GOTO**：`GOTO[from, nonterminal] = to_state`

### 4.3 分析表输出讲解 — 算术表达式文法

终端实际输出的 LR 分析表：

```
State |   +      *      (      )      id     $    ||   E      T      F
---------------------------------------------------------------------------------
    0 |                 s1            s5          ||   2      4      3
    1 |                 s1            s5          ||   6      4      3
    2 |   s7                                acc   ||
    3 |   r4     r4            r4            r4   ||
    4 |   r2     s8            r2            r2   ||
    5 |   r6     r6            r6            r6   ||
    6 |   s7                   s9                 ||
    7 |                 s1            s5          ||          10     3
    8 |                 s1            s5          ||                 11
    9 |   r5     r5            r5            r5   ||
   10 |   r1     s8            r1            r1   ||
   11 |   r3     r3            r3            r3   ||
```

**如何读这张表**（关键讲解）：

| 状态 | + 列 | * 列 | ( 列 | ) 列 | id 列 | $ 列 | 含义 |
|------|------|------|------|------|-------|------|------|
| 0 | · | · | s1 | · | s5 | · | 遇 `(` 移入到 I1，遇 `id` 移入到 I5 |
| 2 | s7 | · | · | · | · | acc | 遇 `+` 移到 I7，遇 `$` 接受 |
| 4 | r2 | s8 | r2 | r2 | r2 | r2 | 遇 `*` 移到 I8，其他按产生式 2 规约（E→T） |
| 5 | r6 | r6 | r6 | r6 | r6 | r6 | 任何终结符都按产生式 6 规约（F→id） |

**这是可解释的输出**：每一格的含义都是确定且唯一的——s 表示 Shift（移入），r 表示 Reduce（规约），acc 表示 Accept（接受），· 表示出错。

### 4.4 TINY 语言分析表

TINY 语言的 LR 分析表更加庞大：20 个终结符列 × 10 个非终结符列 × 44 个状态。但**同样无冲突**，每个表项最多只有一个动作。关键特征：

- I1：`ACTION[1, $] = acc` → 程序接受
- I5：if 语句的入口，通过 `(` → I11 进入条件表达式
- I38：悬挂 else 状态（I38 有 `else → I41` 和 `end → I42` 两条出边）
- I44：`T → if E then S else S end·` 规约

```
✔ 算术表达式文法（扩展 -, /）— 是 SLR(1) 文法，更多状态，0 冲突
✔ TINY 语言文法 — 是 SLR(1) 文法，44 状态，0 冲突
```

### 5.1 LL(1) 分析表构造（额外实现）

为了对照验证，我在 `src/lab2/ll1.rs` 中额外实现了 LL(1) 分析表构造。

**算法**：对每条产生式 `A → α`：
- 若 `FIRST(α)` 不含 ε：对每个 `a ∈ FIRST(α)`，填 `M[A, a] = 该产生式`
- 若 `FIRST(α)` 含 ε：对每个 `b ∈ FOLLOW(A)`，也填 `M[A, b] = 该产生式`
- 如果某格有 >1 条产生式 → LL(1) 冲突

**测试结果**：

测试 1 — `S → +SS | *SS | a`：
```
LL(1) 分析表:
           +          *          a
  S        r1         r2         r3
✔ 该文法是 LL(1) 文法
```
三个产生式的 FIRST 互不相交 → 无冲突。

测试 2 — `T → S, S → (T)TS | ε`：
```
           (          )          $
  S    CONFLICT[2, 3]     r3         r3
✘ 该文法不是 LL(1) 文法，冲突：M[S, (] 中存在多条产生式 [2, 3]
```
`FIRST(S→(T)TS) = {(}` 和 `FOLLOW(S) = {(, ), $}` 在 `(` 上冲突。

## 附一：可能的提问及回答要点

### Q1: 为什么用 BTreeSet 而不是 HashSet？
**A**: BTreeSet 保证元素有序（字典序），输出 FIRST/FOLLOW/项集时排列整齐，便于人工阅读和比对标准答案。LR(0) 项目数量通常不大，O(log n) 与 O(1) 的性能差异可忽略。

### Q2: Rust 不支持继承，如何实现指导书的继承体系？
**A**: 用枚举 `SymbolType { Terminal, NonTerminal, Null }` 区分。终结符和非终结符共用 `GrammarSymbol` 结构体，通过 `sym_type` 字段判别。避免了虚函数调用开销（无 vtable），且更符合 Rust 的数据驱动设计模式。

### Q3: `dependent_in_follow` 的作用？
**A**: 记录 FOLLOW 求解的依赖图。当 `A → αB` 且 α 后的 β 可推导 ε 时，B 的 FOLLOW 依赖 A 的 FOLLOW。在传播阶段统一更新，避免迭代中重复扫描产生式查找依赖关系。这是指导书 `NonTerminalSymbol` 中 `pDependentSetInFollow` 的直接实现。

### Q4: 增广文法的作用？
**A**: (1) 让 DFA 有唯一的起始状态 I0 = `{S' → ·S}`；(2) 统一接受条件——当圆点到达 `S' → S·` 且输入为 `$` 时表示接受；(3) 将所有规约动作统一处理（只有增广产生式规约时才接受）。

### Q5: 为什么闭包用队列而不用递归？
**A**: 队列（BFS）天然避免深度递归导致的栈溢出风险，且 `BTreeSet.insert()` 返回 `false` 即可判断项目是否已存在——已存在的项目不再入队，自动去重。

### Q6: 如何判断 TINY 语言是 SLR(1) 文法？
**A**: 运行 `check_slr1` 返回 `(true, vec![])`——无移入-规约冲突，无规约-规约冲突。分析表填写中的 `conflicts` 也为空。程序输出 `✔ 该文法是 SLR(1) 文法`。

### Q7: 悬挂 else 问题在 TINY 文法中如何体现？
**A**: 状态 I38 同时有 `T → if E then S·end` 和 `T → if E then S·else S end`。在 SLR(1) 下，`end` 和 `else` 不在同一 FOLLOW 集中（`FOLLOW(T→if...end)` 不含 `else`），因此没有冲突。实际上我更倾向于把 else 匹配最近的 if——这符合 SLR(1) 的移入优先策略。

### Q8: HTML 报告中的 Mermaid 图是什么？
**A**: 使用 Mermaid.js 库渲染的状态转移图。每个节点是一个 LR(0) 状态（I0, I1...），边上的标签是驱动文法符。初始状态用蓝色高亮标注。这提供了一个直观的 DFA 可视化。
